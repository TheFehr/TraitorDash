# RFC 7: Secure Session Management & CSRF Protection

## 1. Abstract
This document addresses the security vulnerabilities in the current authentication mechanism of TraitorDash. It outlines the transition from a forgeable, plain-text cookie system to a cryptographically signed session management architecture.

## 2. Current State (The "Debug" Implementation)
The current implementation utilizes a raw `steam_id` cookie:
- **Mechanism:** Upon successful Steam OpenID callback, the `steam_id` is written directly to an unencrypted cookie.
- **Vulnerability:** **Identity Spoofing.** Any user can manually set their `steam_id` cookie to any valid SteamID64 (discoverable via public tools) to gain unauthorized administrative access.
- **Vulnerability:** **No CSRF Protection.** The system currently lacks tokens to prevent Cross-Site Request Forgery on sensitive endpoints like `/api/save`.

## 3. Proposed Architecture: Signed Cookies
To eliminate spoofing while maintaining a lightweight, stateless-first approach, the system will migrate to Hono's `signedCookie` middleware.

### 3.1 Cryptographic Signing
- **Secret Key:** A high-entropy `SESSION_SECRET` will be required in the backend `.env` file.
- **Signing Process:** 
  ```typescript
  // Example implementation
  setSignedCookie(c, 'session', steamId, SESSION_SECRET, {
    path: '/',
    httpOnly: true,
    secure: true,
    sameSite: 'Lax'
  });
  ```
- **Verification:** On every request, the backend will verify the HMAC signature of the cookie. If the user attempts to tamper with the SteamID, the signature verification will fail, and the session will be treated as invalid.

### 3.2 Enhanced Cookie Security
- **HttpOnly:** Prevents client-side scripts from accessing the session cookie, mitigating XSS-based session theft.
- **Secure Flag:** Ensures cookies are only transmitted over HTTPS.
- **SameSite=Lax:** Provides a balance between usability (Steam redirects) and security against CSRF.

## 4. Implementation Plan
1.  **Environment Setup:** Add `SESSION_SECRET` to production and development environments.
2.  **Middleware Update:** Integrate `hono/cookie` signed functions into `auth.ts` and `index.tsx`.
3.  **Migration:** The first time a user with an old, unsigned cookie visits, they will be redirected to re-authenticate via Steam to generate a signed session.

## 5. Future Considerations (JWT & OIDC)
For multi-node or distributed deployments where a shared secret is insufficient, the system can eventually move to JWT-based tokens or a dedicated OIDC provider, though Signed Cookies remain the recommended path for the current single-instance community model.
