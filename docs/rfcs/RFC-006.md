# RFC 6: Secure Command Queue (RCON-less)

## 1. Abstract
This document defines a secure, pull-based method for applying web-based configurations to the Garry's Mod server. By utilizing an HTTP command queue instead of RCON, TraitorDash eliminates the need for exposing sensitive console ports and simplifies deployment in restricted network environments.

## 2. Command Staging & Delivery
TraitorDash operates on an asynchronous "Stage-and-Poll" model.

### 2.1 Web-Side Staging
When an authorized admin submits changes via the dashboard:
1.  **Validation:** The Backend validates every submitted key and value against the extracted JSON Schema (RFC 3).
2.  **White-listing:** Only keys discovered during the Ingestion phase are eligible for updates. This prevents arbitrary command injection.
3.  **Persistence:** Validated updates are converted into a command manifest and stored in the `pending_commands` SQLite table.

### 2.2 Bridge Polling (The "Pull" Model)
Instead of the backend "pushing" to the game server, the GMod Bridge "pulls" its work:
1.  **The Poll:** Every $X$ seconds (default: 5), the Bridge sends a **token-authenticated** GET request to `/api/v1/poll`.
2.  **The Handshake:** The request includes the `TRAITORDASH_TOKEN` sent as a Bearer token in the `Authorization` header.
3.  **The Payload:** The Backend returns a JSON array of all pending commands for that specific server and immediately clears them from the staging table.

## 3. Execution Logic (GMod-Side)
Upon receiving the command manifest, the Bridge executes the updates within the game engine:
*   **ConVar Updates:** Applied using `game.ConsoleCommand` or `RunConsoleCommand(key, value)`.
*   **TTT2 Data Updates:** Applied via internal TTT2 API calls (e.g., `TTT2.DATA.Set`) to ensure database persistence.
*   **Logging:** The Bridge logs the execution of these commands to the server console for auditability.

## 4. Security & Integrity
*   **RCON-less Design:** No RCON password is ever stored or transmitted, significantly reducing the attack surface.
*   **Network Simplicity:** No TCP port-forwarding (27015) is required on the game server's router; as long as the server has outbound internet access, the system functions.
*   **Token Authentication:** Every poll request is authenticated via the unique API token, preventing unauthorized clients from intercepting or spoofing commands.
*   **Auditability:** Every command staged on the web is linked to the authenticated SteamID of the admin, providing a complete record of "who changed what."

## 5. Performance Considerations
*   **Minimal Latency:** While not "instant" (subject to the polling interval), configuration changes do not require the sub-millisecond response time of administrative commands like "Kick" or "Ban."
*   **Batching:** All pending changes are delivered in a single HTTP response, minimizing the number of network round-trips.
