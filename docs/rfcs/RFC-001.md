# RFC 1: TTT2 Dynamic Web Configuration Manager

## 1. Abstract
This document outlines the architecture for an external, web-based configuration panel for Garry's Mod Trouble in Terrorist Town 2 (TTT2) servers. The system is designed to be **100% dynamic**. It automatically detects installed roles (including custom Workshop roles), parses their in-game UI configurations, and automatically generates a mirroring web interface without requiring manual schema updates.

## 2. Motivation
Currently, configuring TTT2 roles requires Server Admins to join the server and navigate the in-game F1 menu. As servers install dozens of custom roles, this becomes tedious and difficult to manage externally. 

By building a web-based UI, admins can configure their servers from a browser. However, hardcoding web forms for every possible TTT2 role is unmaintainable. Therefore, this project implements a "headless execution" pipeline to automatically extract UI definitions directly from the server's live `.lua` files.

## 3. System Architecture
The system consists of three distinct boundaries:
1. **The GMod Server (Source Engine):** Runs a lightweight Lua script to export files and poll state.
2. **The Web Backend (Node.js):** Acts as the bridge. It parses raw Lua code in a mocked VM, maintains the database, and stages commands for the server to pull.
3. **The Web Frontend (React / Vue / Svelte):** A dynamic rendering engine that builds HTML forms based on JSON schemas.

### 3.1 The "Automagic" Schema Pipeline (Initialization)
To guarantee the Web UI always matches the server's installed mods, the system uses a mocked Lua environment to extract data.
1. **File Fetching:** On server `Initialize`, a custom GMod server-side script reads all `.lua` files in `terrortown/entities/roles/*.lua`.
2. **Payload Transmission:** GMod POSTs these raw Lua strings to the Web Backend (`/api/ingest`).
3. **Headless Execution:** The Backend spins up a Lua VM (e.g., Fengari) containing a mocked GMod environment (`vgui`, `ROLE`, `CLIENT = true`).
4. **Metatable Interception:** A generic `fakeForm` object utilizes a Lua metatable. Any method called on it starting with `Make` (e.g., `MakeCheckBox`, `MakeSlider`, `MakeComboBox`) is intercepted. The method name and its parameter table are extracted and converted into a generic JSON schema.
5. **Schema Storage:** The resulting JSON structure is saved to the backend database, dictating exactly which controls the Frontend should render.

### 3.2 Bidirectional State Synchronization
The UI schema defines *what* controls exist, but the system must also track the *current values* of those ConVars and apply new ones.

**Reading Data (GMod -> Web):**
*   GMod cannot act as an HTTP server, so it must poll.
*   A timer runs on the GMod server (e.g., every 10-30 seconds), collecting the current integer/string values of all registered TTT2 ConVars.
*   GMod sends an HTTP POST payload of these current states to the Web Backend.
*   The Backend updates its database, ensuring the Web Frontend sliders/checkboxes accurately reflect live server data.

**Writing Data (Web -> GMod):**
*   When an Admin clicks "Save" on the Web UI, an HTTP POST is sent to the Backend.
*   The Backend stages these updates in a **Secure Command Queue** (as defined in **RFC 006**).
*   The GMod server pulls these commands via a periodic HTTP GET request.
*   The change applies to the game server within seconds of being pulled.

## 4. Security & Authentication Considerations
*   **Backend Authentication:** The Web UI must sit behind an admin login system (e.g., Steam OpenID).
*   **Token Authentication:** Every request from the GMod Bridge is authenticated via a unique API token stored on the server.
*   **Payload Validation:** The Web Backend must sanitize inputs before staging them in the command queue to prevent command injection.

## 5. Technical Stack Recommendations
*   **Backend:** Node.js (Bun / Hono)
*   **Lua VM:** Fengari (Runs standard Lua 5.3 inside Node.js)
*   **Frontend:** React (TSX) with HTMX and Alpine.js for lightweight interactivity.
*   **Database:** SQLite (using `bun:sqlite`).

## 6. Open Questions / Next Steps
1.  **Determine Frontend Framework:** Finalize which JS framework will be used to render the dynamic forms.
2.  **Mock Environment Completeness:** Which global GMod functions (e.g., `Color()`, `CreateConVar()`) need empty dummy functions in the Fengari VM to prevent execution crashes during schema extraction?
3.  **UI Component Mapping:** Map all known TTT2 `Make*` form elements to HTML/CSS equivalents (Checkbox -> Toggle Switch, Slider -> Range Input, Binder -> Keystroke Listener).
