# RFC 4: The Dynamic Web Dashboard

## 1. Abstract
The TraitorDash Frontend is a high-performance, server-side rendered (SSR) interface built on **Bun** and **Hono**. It utilizes **HTMX** for partial page interactivity and **Alpine.js** for lightweight client-side state, ensuring a responsive, "app-like" experience while maintaining a server-centric architecture and a thin client footprint.

## 2. Technical Stack
*   **Runtime:** Bun (Native TypeScript execution).
*   **Web Framework:** Hono (Fast, lightweight, JSX/TSX support for type-safe SSR).
*   **Interactivity:** HTMX (AJAX-based partial HTML swaps for search and navigation).
*   **Client Logic:** Alpine.js (Lightweight tracking for "Dirty" form states and UI toggles).
*   **Styling:** Tailwind CSS (Modern, utility-first styling).

## 3. UI Components & Interactivity

### 3.1 Searchable Sidebar (HTMX)
*   The search input utilizes `hx-get="/sidebar?q=..."` with a `keyup changed delay:300ms` trigger.
*   The Bun backend filters the role list based on the query and returns a partial HTML fragment of the sidebar.
*   HTMX swaps the old list with the new one, providing a "live search" feel with zero client-side search logic.

### 3.2 "Dirty" State Tracking (Alpine.js)
*   The settings form utilizes an Alpine `x-data` object to store initial values during the initial render.
*   Logic compares current input values to the initial state.
*   If a difference is detected, a Tailwind-styled "Save Changes" footer/toast is toggled visible, mirroring the TTT2 "Apply" button behavior.

### 3.3 Dynamic Role Panels (SSR)
*   The backend maps the JSON schema extracted in RFC 3 to server-side TSX components.
*   **Standard Components:**
    *   **Toggle (MakeCheckBox):** Standard checkbox with Tailwind styling.
    *   **Slider (MakeSlider):** HTML5 range input with numeric feedback.
    *   **Select (MakeComboBox):** Styled dropdown menu.
    *   **Text (MakeTextEntry):** Single-line string input.
    *   **Keybind (MakeBinder):** Interactive keystroke listener.

## 4. Admin Presence & Safety
*   The backend checks the "Active Admins" state from the Bridge (RFC 2) during request rendering.
*   If admins are detected, a persistent warning banner is injected: *"Attention: Admin [Name] is online. In-game changes take priority and may overwrite web-based edits."*

## 5. Deployment & Build
*   **No-Build Workflow:** Bun executes TSX files directly, removing the need for a separate compilation step for the backend.
*   **Assets:** Tailwind CSS is processed once and served as a static file. HTMX and Alpine.js are served via CDN or bundled as lightweight vendor scripts.
