# RFC 2: The TraitorLink Bridge Plugin

## 1. Abstract
The **TraitorLink Bridge** is a server-side Lua plugin for Garry's Mod designed to facilitate bidirectional communication between a TTT2 game server and the TraitorDash Web Backend. Its primary goal is to provide a "Dumb Bridge" that extracts role configurations and synchronizes live server state with minimal performance impact.

## 2. Discovery & Ingestion Pipeline
To ensure 100% dynamic support for all TTT2 roles, the bridge utilizes TTT2's internal registration systems rather than static file scanning.

### 2.1 Role & Asset Discovery
The bridge hooks into `TTT2FinishedLoading` to ensure all addons have registered their roles.
1.  **Iterate Roles:** Loop through `roles.GetList()`.
2.  **Trace Source:** Use `debug.getinfo(role.PreInitialize).source` to locate the absolute path of the defining `.lua` file.
3.  **Localization Scraping:** Recursively search the addon's root directory (identified from the role path) for `lang/en/*.json` or `lang/en/*.lua` files.
4.  **State Snapshot:** Gather the current values of all registered ConVars and `TTT2.DATA` entries.

### 2.2 MD5 Caching & Performance
To minimize network overhead and prevent map-change hitches:
1.  **Handshake:** Upon initialization, the bridge sends a list of filenames (roles and translations) and their MD5 hashes to the backend.
2.  **Differential Sync:** The backend compares these hashes against its database and responds with a "Dirty List" of files it actually needs.
3.  **Transmission:** The bridge only POSTs the content of files on the "Dirty List," along with the full **State Snapshot**.

## 3. Bidirectional State Synchronization

### 3.1 Data Tracks
TTT2 settings exist in two distinct domains, both of which the bridge must manage:
*   **ConVar Track:** Standard console variables (e.g., `ttt_hitman_health`).
*   **TTT2.DATA Track:** Persistent settings stored in the server's SQLite database (`sv.db`) (e.g., role enabled toggles, probabilities).

### 3.2 GMod -> Web (Live Updates)
*   **ConVars:** The bridge uses `cvars.AddChangeCallback` to detect in-game changes.
*   **Internal Data:** The bridge hooks into TTT2 data update events to capture changes to non-ConVar settings.
*   **Payload:** Changes are batched and sent via HTTP POST to the backend with a high-resolution timestamp.

### 3.3 Web -> GMod (Remote Execution)
The backend applies changes via an RCON connection:
*   **ConVar Updates:** Executed directly via standard RCON (e.g., `ttt_hitman_speed 1.5`).
*   **Data Updates:** Executed via a custom bridge command: `traitordash_setdata <scope> <key> <value>`. This command is restricted to the server console (`FCVAR_SERVER_CAN_EXECUTE`).

## 4. Concurrency & Safety
*   **Server-First Authority:** In the event of a race condition, the value set in-game (GMod) is treated as the authoritative state.
*   **Presence Detection:** The bridge monitors for active admins (players with `superadmin` or specific TTT2 permissions who are currently active). The sync payload includes an `active_admins` list.
*   **Session Locking:** If an admin is detected, the Web UI displays a "Session Lock" warning, notifying the web user that in-game changes are currently prioritized and may overwrite web-based edits.
*   **Sanitization:** All inputs received via the `traitordash_setdata` command are type-validated (e.g., ensuring a boolean is strictly 0 or 1) before being applied to the game state.

## 5. Security
*   **Token Authentication:** Every HTTP request from the bridge includes a `TD-Key` header, a unique UUID defined in the server's local configuration.
*   **Transient Code Processing:** The backend is mandated to treat transmitted Lua code as ephemeral; it is used for schema extraction and immediately discarded to respect third-party licensing.
