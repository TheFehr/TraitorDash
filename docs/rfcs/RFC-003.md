# RFC 3: Headless Lua Mocking & Schema Extraction

## 1. Abstract
This document defines the backend pipeline for transforming raw Lua source code into a JSON-based UI Schema. This transformation is achieved by executing role files within a mocked Garry's Mod environment inside a Node.js-based Lua VM (Fengari).

## 2. Execution Environment (Fengari VM)
To execute GMod Lua code safely and extract its UI structure, the backend implements a "padded" environment.

### 2.1 GMod Emulation (Mocks)
Before executing a role file, the VM is injected with global mocks:
*   **Environment Flags:** `CLIENT = true`, `SERVER = false`.
*   **Primitives:** `Color()`, `Vector()`, `Angle()` (returning simple table equivalents).
*   **ConVar Access:** `GetConVar(name)` is mocked to return values from the **State Snapshot** provided by the Bridge during ingestion. This state is extracted by the bridge scanning the role source code for `GetConVar` calls and fetching their current values. This enables support for conditional UI logic (e.g., `if GetConVar("..."):GetBool() then ... end`).
*   **Localization:** `LANG` methods are mocked to return values from an internal dictionary. This dictionary is populated because the Bridge concatenates the contents of relevant `lang/en/*.lua` files to the end of the role script before transmission.

### 2.2 Tracing & Metatable Proxy
The `parent` object passed to `ROLE:AddToSettingsMenu(parent)` is a Lua metatable proxy ("Spy Object").
*   **Interception:** Any method call starting with `Make*` (e.g., `MakeCheckBox`, `MakeSlider`) is intercepted.
*   **Serialization:** The method name and its arguments (the parameter table) are captured and added to an internal `extracted_ui` list.
*   **Graceful Degradation:** Unrecognized method calls on the proxy are logged but ignored, preventing crashes during the extraction of files using non-standard UI libraries.

## 3. The Extractor Registry (Plugin Architecture)
The backend uses a plugin-based registry to interpret intercepted calls. This allows for modular support of different UI elements without modifying the core extraction engine.

### 3.1 V1 Support (Standard TTT2)
The initial registry includes extractors for standard `DFormTTT2` methods:
*   `MakeCheckBox` -> Web Toggle
*   `MakeSlider` -> Web Range Input
*   `MakeComboBox` -> Web Select/Dropdown
*   `MakeTextEntry` -> Web Text Input
*   `MakeBinder` -> Web Key-Listener

### 3.2 Extensibility
Third-party developers can register custom extractors to support non-standard UI libraries or specific role-based GUI elements by mapping Lua method signatures to Web Component definitions.

## 4. Schema Generation & Localization
1.  **Extraction:** The results of the `extracted_ui` list are converted to a JSON schema.
2.  **Localization Mapping:** Any string value in the schema that matches a key in the ingested `langs` files is flagged for client-side translation.
3.  **Caching:** The final JSON schema is cached against the MD5 hash of the role file. Extraction only re-runs if the file content changes.

## 5. Security Considerations
*   **VM Isolation:** Each extraction run occurs in a fresh, ephemeral VM to prevent state leakage between different roles or servers.
*   **Resource Limits:** The VM is execution-time limited to prevent malicious or poorly written Lua code from causing infinite loops or excessive CPU usage on the backend.
