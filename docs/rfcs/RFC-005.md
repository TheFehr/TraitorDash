# RFC 5: Identity & Multi-Server Management

## 1. Abstract
This document outlines the authentication and server management architecture for TraitorDash. It utilizes **Steam OpenID** for identity verification and implements a "Zero-Config" permission model that synchronizes administrative rights directly from the Garry's Mod server via the **Common Admin Mod Interface (CAMI)**.

## 2. Authentication: Steam OpenID
*   **Identity Provider:** Users authenticate via the official Steam OpenID 2.0 endpoint.
*   **Identification:** Users are uniquely identified by their **SteamID64**.
*   **Super Admins:** A list of "Owner" SteamIDs is defined in the backend `.env` file (`SUPER_ADMINS="id1,id2"`). These users have absolute authority to add/remove servers and manage global dashboard settings.

## 3. Automated Permission Synchronization (CAMI)
To eliminate manual user management, TraitorDash leverages existing in-game permissions.
*   **Permission Scraping:** The Bridge (RFC 2) periodically checks which SteamIDs currently have the `ttt2_conf_edit` privilege via CAMI.
*   **Authorization Sync:** These authorized SteamIDs are sent to the Web Backend during the State Sync phase and cached in the local database.
*   **Dynamic Access:** A user's ability to edit a server's configuration on the web is automatically granted or revoked based on their current rank/permissions on the GMod server.

## 4. Multi-Server Management
*   **Server Identity:** Each GMod server is registered in the dashboard with a unique name and RCON credentials.
*   **API Tokens:** The backend generates a unique `TRAITORDASH_TOKEN` (UUID) for each server. This token must be present in the Bridge's HTTP headers for the backend to accept data.
*   **Read-Only Guest View:** Any user logged in via Steam can view the status (Map, Players, Current Round) of registered servers, but configuration panels remain locked unless authorized.

## 5. Persistence (`bun:sqlite`)
The backend uses Bun's native SQLite driver for lightweight data storage.
### 5.1 Data Schema
*   **`servers`**: `id`, `name`, `api_token`, `rcon_ip`, `rcon_port`, `rcon_password`.
*   **`authorized_admins`**: `server_id`, `steam_id`, `last_seen` (Synchronized from CAMI).
*   **`audit_log`**: `id`, `server_id`, `steam_id`, `action`, `old_value`, `new_value`, `timestamp`.

## 6. Privacy & Tenancy
*   **Private-First:** TraitorDash is designed as a private instance for a single community or friend group.
*   **OpenID Security:** The backend validates the return URL and nonce of every Steam OpenID response to prevent replay attacks or spoofing.
